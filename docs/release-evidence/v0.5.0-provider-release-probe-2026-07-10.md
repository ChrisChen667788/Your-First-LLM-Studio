# v0.5.0 Provider Release Probe Evidence

Date: 2026-07-10

## Purpose

Provider Ops previously counted only user chat traffic. A real connection check already makes a minimal remote `models` and `chat/completions` request, but that proof was not visible to the promotion gate.

This change introduces a distinct `release-probe` evidence purpose. It remains separate from user chat traffic, token accounting, and estimated provider cost.

## Contract

- `POST /api/admin/provider-health/evidence` accepts `{ "action": "run-release-probe", "targetId": "..." }`.
- The probe runs only for a configured remote OpenAI-compatible target.
- Missing credentials fail before any outbound request and do not create a successful evidence record.
- A completed probe is saved in connection-check history with `evidencePurpose: "release-probe"` and creates a Provider event in the Experiments timeline.
- `provider.ops-evidence-summary.v1` reports probe totals, successful probe totals, latest timestamps, and per-target probe state.
- Promotion passes the Provider Ops source only when the current 24 hour window has either a normal remote chat request or a successful release probe, while auth, quota, and policy blockers still hold the gate.

## Operator Flow

1. Open `/admin` and find Provider usage / health desk.
2. Choose a configured remote target in Release probe.
3. Run the probe and inspect the success count, timestamp, connection-check history, and Experiments Provider event.
4. Re-read `/api/experiments/promotion-gate`; it should only improve after the remote check itself returns success.

## Verification

```bash
npm run typecheck:changed
SMOKE_CURL_MAX_TIME=90 npm run smoke:routes
curl -fsS http://localhost:3011/api/admin/provider-health/evidence?windowHours=24
```

The route smoke contract requires the `releaseProbe.totalCount` and `releaseProbe.successCount` fields. It does not invoke POST, so automated smoke runs do not spend provider quota.
